about

I build software on both sides of the fence — shipping features and trying to break them. Formal training gave me Laravel and PHP for the web, C# and Java for desktop, Java/Kotlin for Android. Outside class, I picked up Python, C/C++, and Linux over the past 6+ years, largely by pointing them at ESP32 hardware and wireless protocols.

The two sides feed each other more than people expect — writing the API makes me better at breaking one, and breaking enough of them has made me a more careful engineer. Most of what I know about security came from CTFs (HackTheBox, TryHackMe) and building my own tools when the existing ones didn't fit what I needed.

Day to day that's a mix of client projects in Laravel/C#, personal embedded builds on ESP32, and security practice in a home lab — all logged below. Linux Mint is my daily driver, with Docker, Apache/Nginx, and MariaDB for local dev; deployment has ranged from a Raspberry Pi 3B+ running Nginx to full VPS setups, depending on what the project needs.

self-taught

security & offensive tooling

This started as "how do I break the apps I build" and grew from there. Most of it is self-taught and hands-on: working through OWASP Top 10 categories against my own projects, practicing on HackTheBox and TryHackMe, and building/breaking tools on ESP32 hardware. I keep a no-root, non-jailbroken Android phone running Termux as a deliberate constraint — if a technique only works with root, I haven't really learned it yet.

I'm comfortable with network penetration testing, firewall evasion and pivoting, firmware reverse engineering, and exploit development — mostly API and injection vulnerabilities, since that's where my dev background gives me an edge. Binary/low-level exploitation is the area I'm still actively building depth in. All of this is practiced in my own lab, on CTF platforms, or with explicit authorization — never against systems I don't have permission to test.

When a tool I need doesn't exist, I build it — NRSuite, DuckESP, ESP-Bridge, Termux-ESP-Flasher, and BridgeProtocol all started that way. I've also contributed to Bruce, an established open-source red-team firmware project for ESP32 devices (AGPL-licensed, built for authorized security testing).

Termux is basically a second daily driver at this point — I've pushed it to the point of running full workflows without ever touching a laptop, which usually surprises people until they see it.

Two findings reported through private/responsible disclosure to a nationwide retail-tech vendor. Details below are deliberately high-level — no exploit code, endpoints, or vendor name — out of respect for the disclosure agreement:

  • Unauthenticated admin account creation vulnerability (2021) — reported and acknowledged by the vendor's team
  • Critical remote code execution via command injection, leading to full system compromise with root privileges (2022) — reported privately, patched, no public details disclosed
